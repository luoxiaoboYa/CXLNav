# CXSearch 共享与推荐逻辑设计

日期：2026-04-24

## 目标

明确“个人收藏”“共享”“推荐发现”的边界，避免插件推荐、用户共享和公共推荐池混淆。

## 核心结论

采用三层模型：

```text
我的站点 → 共享申请 → 推荐发现
```

- 我的站点：用户私人收藏库，只对自己可见。
- 共享申请：用户在主站申请把自己的某个收藏站点公开出去。
- 推荐发现：系统推荐和审核通过的用户共享内容组成的公共展示页。

一句产品规则：共享是用户发起公开的动作，推荐是平台对外展示的结果。

## 个人收藏

个人收藏用于解决用户自己的站点管理问题。

用户可以通过主站或浏览器插件保存站点到“我的站点”。个人收藏默认只对自己可见，不直接进入公共推荐池。

个人收藏允许相对宽松，但需要记录安全状态：

- 未校验
- 校验中
- 安全
- 风险
- 无法访问

风险站点可以保留在个人收藏中并给出提示，但不能发起共享。

## 插件边界

浏览器插件只做快速私人收藏。

插件可以做：

- 右击保存当前页面到我的站点。
- 自动获取当前页面标题和链接。
- 手动补充分类、书签路径、普通标签和详情。
- 同步到用户的私人站点库。

插件不做：

- 不直接共享到推荐发现。
- 不直接提交公共推荐。
- 不直接进入审核流程。
- 不让未确认整理的站点公开展示。

如果用户希望公开某个插件收藏的站点，需要回到主站，在我的站点中整理后发起共享申请。

## 共享申请

共享申请只能从主站发起，且必须基于“我的站点”中的已有站点。

流程：

```text
我的站点详情 / 编辑页
→ 发起共享申请
→ 自动校验
→ 后台审核
→ 审核通过后进入推荐发现
```

共享申请应保留与个人站点的关联，但公共展示使用独立的共享记录，避免直接暴露用户的私人收藏数据。

共享申请需要的信息：

- 标题
- URL
- 描述 / 推荐理由
- 分类
- 普通标签
- 来源用户
- 来源个人站点 ID
- 提交时间
- 当前状态
- 审核结果或失败原因

## 校验与审核

共享申请必须遵循 docs/superpowers/specs/2026-04-24-url-safety-design.md 中的 URL 安全与爬取防护规则。链接有效性校验不等于安全校验，安全状态未通过的站点不得进入后台审核和推荐发现。


共享申请公开前需要两层处理。

第一层是自动校验：

- URL 格式是否合法。
- 协议是否只允许 http / https。
- 链接是否可访问。
- 是否重复提交。
- 标题、描述、分类、普通标签是否完整。
- 是否存在明显安全风险。

第二层是后台审核：

- 内容是否适合公开展示。
- 是否疑似广告、钓鱼、恶意下载或违法违规内容。
- 描述和标签是否准确。
- 是否应该合并到已有推荐内容。

审核通过后才进入推荐发现页。审核失败需要给出原因，供用户修改或放弃。

## 推荐发现

推荐发现页只展示公共可见内容，包括：

- 系统推荐：平台预置或运营维护的优质站点。
- 用户分享：通过共享申请并审核通过的用户收藏站点。

推荐发现不直接展示用户私人收藏，也不展示未审核的共享申请。

热门推荐不是独立来源，而是推荐发现中基于访问量、收藏量、点赞量等指标生成的排序或标记。

## 去重规则

如果用户共享的 URL 已经存在于推荐发现中，不直接创建重复推荐项。

可采用：

- 关联到已有推荐项。
- 增加分享人数、收藏人数或推荐权重。
- 允许用户补充自己的推荐理由，但公共主记录保持唯一。

## 权限规则

- 未登录用户：只能查看系统推荐。
- 登录用户：可以查看系统推荐和已公开的用户分享内容。
- 登录用户：可以从自己的站点发起共享申请。
- 管理员 / 后台角色：可以处理共享审核、下架公共推荐内容。

## 关键边界

- 插件不发起共享，只保存私人收藏。
- 用户不能直接创建公共推荐内容，只能提交共享申请。
- 推荐发现只展示系统推荐和审核通过的共享内容。
- 共享记录与私人收藏记录分离，避免误公开私人数据。
- 链接有效性校验不是审核，安全校验和内容审核都需要保留。
