# 后期需求备忘：多身份书签列表

日期：2026-04-28

## 背景

用户可能同时拥有多种身份和多个浏览器账号，例如：

- A 账号用于工作。
- B 账号用于私人生活。
- 不同账号下的浏览器书签并不相同。

在实际使用中，用户可能正在工作账号环境里，但临时需要打开私人账号书签里的某个站点；或者正在私人环境里，需要查找工作账号之前导入过的资料。如果必须切换浏览器账号或重新登录浏览器，会打断当前工作流。

## 需求想法

在 CXSearch 中支持一个用户创建多个独立的书签列表。每个列表可以对应一种身份、一个浏览器账号、一个使用场景或一批导入来源。

用户可以把不同浏览器账号导出的书签分别导入到不同列表中，再在 CXSearch 站点内统一查看、搜索和打开。这样即使当前浏览器没有登录对应账号，也可以快速访问之前导入到 CXSearch 的那组书签。

## 初步用户场景

- 工作列表：导入工作账号浏览器书签，包含项目、后台、文档、协作工具。
- 私人列表：导入私人账号浏览器书签，包含生活、学习、娱乐、个人工具。
- 临时/专题列表：为某个项目、学习主题、旅行计划单独导入或整理一组书签。

用户在任意设备或任意浏览器账号下登录 CXSearch 后，可以切换列表并搜索对应书签，不必切换浏览器账号。

## 初步功能范围

- 用户可以创建、重命名、删除多个书签列表。
- 导入书签时可以选择导入到哪个列表。
- “我的站点”支持按列表切换或筛选。
- 搜索可以选择当前列表、多个列表或全部列表。
- 每个列表可以保留自己的分类、书签路径、标签和导入来源信息。
- 后续浏览器插件保存当前页面时，可以让用户选择保存到哪个列表，或使用默认列表。

## 价值

- 支持多身份、多账号、多场景管理，贴近真实使用习惯。
- 降低用户切换浏览器账号的成本。
- 让 CXSearch 从“单一个人书签库”扩展为“统一书签中枢”。
- 为后续行为统计、使用偏好、插件同步预留更清晰的数据边界。

## 需要后续细化的问题

- 列表之间是否允许同一 URL 重复存在，还是提供跨列表合并建议。
- 分类、标签、书签路径是全局共享，还是每个列表独立维护。
- 首页常用站点默认展示哪个列表，是否需要“当前工作区/当前列表”概念。
- 浏览器插件的默认保存列表如何设置。
- 分享到推荐库时，是否需要暴露来源列表信息；初步倾向不暴露。
- 用户行为统计按用户汇总，还是按列表分别统计。

## 暂不进入当前 MVP

当前阶段只记录需求，不立即实现。MVP 仍以单个“我的站点”个人库打通核心数据流为主。
